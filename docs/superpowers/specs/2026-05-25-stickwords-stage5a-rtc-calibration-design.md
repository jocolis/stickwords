# StickWords Stage 5A RTC Calibration Design

## Goal

Give the M5Stick C Plus a trustworthy local clock by setting and reading its built-in BM8563 RTC.

This stage is intentionally small. It does not implement offline due-card scheduling and does not change the review algorithm. It only creates a verifiable time foundation for later offline behavior and auto power-off work.

## Current Problem

The firmware currently treats the PC backend's `generated_at` value as a review timestamp source, but it does not write that timestamp into the M5Stick RTC.

As a result, when Wi-Fi is unavailable, the device can reuse the most recently synced due-card batch, but it cannot know the real current date and time. That blocks future behavior such as:

- deciding whether cached future cards are now due
- showing meaningful local time in diagnostics
- validating that power-off and restart preserved time

## Hardware And Library Facts

M5Stick C Plus includes a BM8563 RTC. The local `M5StickCPlus` Arduino library exposes:

- `M5.Rtc.GetTime(...)`
- `M5.Rtc.GetDate(...)`
- `M5.Rtc.SetTime(...)`
- `M5.Rtc.SetDate(...)`

The local AXP192 library documents `PowerOff()` as cutting all power except `LDO1 (RTC)`. This means RTC time should continue while the device is powered off, as long as the battery is not exhausted and RTC power remains available.

## Scope

In scope:

- Parse the backend `generated_at` timestamp already returned by `/api/device/tasks`.
- Write valid UTC timestamp values into BM8563 after a successful task sync.
- Read RTC date/time at boot and after sync.
- Mark RTC as valid only when the year is plausible.
- Log RTC state clearly over serial for real-device validation.
- Update handoff/dev-log notes with the real-device validation procedure.

Out of scope:

- Offline selection of due cards.
- Persisting full per-card scheduling state on the M5Stick.
- Time zone settings or local-time display. Stage 5A should use UTC internally.
- RTC alarm wakeup.
- Auto power-off. That will be Stage 5B.

## Timestamp Rules

The backend currently returns ISO-like UTC strings such as:

```text
2026-05-24T09:30:00Z
```

The firmware should accept the simple UTC shape it already receives:

```text
YYYY-MM-DDTHH:MM:SSZ
```

If parsing fails, the firmware should not change RTC. It should log the parse failure and continue the existing sync behavior.

The firmware should consider RTC valid when:

- year is at least `2024`
- month is `1..12`
- date is valid for that month, including leap-year handling for February
- hour is `0..23`
- minute and second are `0..59`

This is still intentionally bounded to the timestamp shape used by the backend, but it should reject impossible calendar dates before writing the RTC.

## Firmware Behavior

On boot:

1. Initialize M5Stick as today.
2. Read BM8563 RTC.
3. Print a serial line like:

```text
RTC now=2026-05-25T08:10:03Z valid=1
```

If the value is not plausible:

```text
RTC now=invalid valid=0
```

After a successful `/api/device/tasks` response:

1. Parse `generated_at`.
2. If valid, call `M5.Rtc.SetDate(...)` and `M5.Rtc.SetTime(...)`.
3. Read back the RTC and log the resulting time:

```text
RTC set=2026-05-25T08:11:00Z
RTC now=2026-05-25T08:11:00Z valid=1
```

If parsing fails:

```text
RTC set skipped: invalid generated_at
```

## Backend Behavior

No new endpoint is required in Stage 5A.

The existing `/api/device/tasks` response already includes `generated_at`, generated by the PC backend's clock. The firmware will use that value as the RTC calibration source.

## Real-Device Validation

Validation procedure:

1. Start the PC backend.
2. Boot the M5Stick with Wi-Fi available.
3. Confirm serial output shows `RTC set=...` and `valid=1`.
4. Power off the M5Stick.
5. Wait at least 1 to 2 minutes.
6. Boot again and confirm the serial `RTC now=...` time moved forward.

Optional follow-up validation:

1. Disable Wi-Fi or stop the backend.
2. Boot the M5Stick.
3. Confirm the boot log still shows a plausible RTC time even though sync fails.

## Risks And Limits

- RTC can become invalid if the battery is fully exhausted.
- The RTC may drift over time; it should be recalibrated whenever backend sync succeeds.
- Stage 5A does not make offline due-card decisions. It only proves the device has time.
- The current firmware JSON parser is small and only supports the known backend shape; timestamp parsing should stay similarly bounded.

## Future Work

- Stage 5B: auto power-off after 3 minutes of no interaction.
- Stage 5C: local scheduling cache with `due_at` and review-state metadata so M5Stick can select due cards offline using RTC.
